# storage/

This directory stores local runtime evidence for the in-process DFS topology.

## Paxos Log

`paxos_log.txt` records the simplified Paxos message flow generated by CLI and DFS operations.

Mutating DFS operations produce log entries such as:

```text
Node 1: PROPOSE touch example.txt, proposal #1
Node 1: ACCEPT from leader 1 touch example.txt, proposal #1
Node 2: ACCEPT from leader 1 touch example.txt, proposal #1
Node 3: ACCEPT from leader 1 touch example.txt, proposal #1
Node 1: LEARN touch example.txt, proposal #1
Node 2: LEARN touch example.txt, proposal #1
Node 3: LEARN touch example.txt, proposal #1
Node 1: COMMIT touch example.txt, proposal #1, majority 3/3
```

These entries provide correctness evidence for the required protocol concepts:

- fixed leader
- proposal number
- `ACCEPT` messages
- `LEARN` messages
- majority-based commitment with 3 replicas

## Storage Model

The current implementation keeps Chord node state in memory while the CLI or tests are running. Metadata and page ownership are inspectable through DFS commands such as:

```text
dfs> stat example.txt
```

The `stat` output includes page descriptors with:

- `page_no`
- `guid`
- Chord owner node ID
- intended replica node IDs

Durable disk recovery is not implemented in this version. The persistent artifact in this directory is the Paxos evidence log.

## Fresh Log Runs

The Paxos log is append-only during normal execution. To collect a fresh log before a test/demo run, remove it first:

```powershell
Remove-Item storage/paxos_log.txt
```

The file will be recreated automatically when the next DFS update commits.
